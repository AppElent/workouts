## Summary

The host-a-session feature works on the happy path (create → open → guest/host scores → leaderboard sorts correctly → close), but several behaviors are not as intended: scores/prescriptions are modeled per-WOD rather than per-movement, signed-in athletes are indistinguishable on the leaderboard, submissions duplicate freely and pollute the athlete's WOD library, several backend errors dump raw Convex stack strings to the user, and closing a session strands participants' active workout sessions (locking them out of the rest of the app). Counts: 1 blocker, 9 major, 7 minor.

## Action Items

### Blockers

- [ ] **Closing (or leaving) a hosted workout strands the participant's active workout session, locking them out of the app**
  - **What:** Joining a hosted workout creates a `workoutSessions` row with `status: 'active'` for the participant (and for the host in `hostAndParticipate`). `close` only patches the hosted workout's status; it never finishes those sessions. A participant/host is then permanently left with an active session, which the app treats as the single allowed active session. Reproduced: after closing "Review Test WOD", the test user still had its participant session active, and the next `open` (host-and-participate) failed with "Finish or cancel your active workout before opening." The user must manually hunt down and cancel/finish the stranded session before they can join/open/start any workout again.
  - **Where:** `/hosted-workouts/$id` → [convex/hostedWorkouts.ts:306](convex/hostedWorkouts.ts) (`close`), interacting with the active-session guards in [convex/hostedWorkoutParticipants.ts:31](convex/hostedWorkoutParticipants.ts) (`join`) and [convex/hostedWorkouts.ts:278](convex/hostedWorkouts.ts) (`open`). Sessions are created in `join` ([convex/hostedWorkoutParticipants.ts:40](convex/hostedWorkoutParticipants.ts)) and `open` ([convex/hostedWorkouts.ts:287](convex/hostedWorkouts.ts)).
  - **Type:** bug
  - **Fix direction:** When a host closes a hosted workout, finish (set `status: 'completed'`, set `endTime`) every participant `workoutSessions` row linked via `hostedWorkoutParticipants` for that hosted workout (including the host's own). Alternatively/additionally, decouple hosted participation from the "one active session" constraint so a hosted session never blocks a user's normal logging. Pick one and make it consistent across `join`/`open`/`close`.
  - **Acceptance:** After a host closes a hosted workout, no participant (nor the host) is left with an `active` session tied to it; each such user can immediately start/join another workout without seeing "Finish or cancel your active workout…". Verify by: join as a participant, host closes, then that participant can start a normal workout with no error.

### Major

- [ ] **Weights/prescriptions are per-WOD, not per movement (item) — the user's primary flag**
  - **What:** A WOD block's `load` score is a single `load` + `loadUnit` for the whole block, and the builder captures only one free-text "prescription" line per level (stored as `movements[0].name`). There is no way to specify a weight per movement/item. The `load` score form shows one "LOAD" stepper for the entire WOD (confirmed on the "Grace" load WOD). The schema's `movement` object already has `reps`/`weight`/`unit`/`distance` fields, but the builder never populates them and scoring ignores them. Intended: weights should be settable per item in the WOD.
  - **Where:** builder `/hosted-workouts/new` → [src/components/hosted/HostedWorkoutBuilder.tsx:275-313](src/components/hosted/HostedWorkoutBuilder.tsx) (single `movements[0]` input per level, no weight field); score form `/log/$sessionId` & `/join/$token` → [src/components/hosted/HostedScoreForm.tsx:182-202](src/components/hosted/HostedScoreForm.tsx) (single `load`); model in [convex/schema.ts:174](convex/schema.ts) (`hostedWorkouts.template.wodBlocks[].levels[].movements[]`) and [convex/hostedWorkouts.ts:19](convex/hostedWorkouts.ts) (`movement` validator already supports per-movement `weight`/`unit`).
  - **Type:** bug (core intent)
  - **Fix direction:** Confirm the exact intended shape with the design, then: (a) let the builder add multiple movements per level, each with a name + optional weight/reps/unit; (b) persist those into the existing `movements[]` structure (weights already fit the schema); (c) decide whether scoring should stay a single aggregate result or capture per-movement loads, and adjust `HostedScoreForm`/`hostedWorkoutSubmissions` accordingly. At minimum, per-movement weights must round-trip builder → template → session plan/leaderboard.
  - **Acceptance:** A host can author a WOD level with ≥2 movements each carrying its own weight; those weights persist and are shown to participants on `/join/$token` and `/log/$sessionId`; scoring reflects the intended (per-item vs aggregate) model without losing the per-movement weights.

- [ ] **Signed-in participants have no display name — everyone shows as "Signed-in athlete"**
  - **What:** `JoinHostedWorkout.handleJoin` calls `join({ token })` with no `displayName`, so every signed-in participant is stored nameless and renders as the generic "Signed-in athlete" on the participants list and every leaderboard row. Guests are *required* to enter a 2–80 char name, but signed-in athletes can't provide one at all, so multiple signed-in athletes are indistinguishable. Reproduced: two signed-in submissions both showed as "Signed-in athlete".
  - **Where:** `/join/$token` → [src/components/hosted/JoinHostedWorkout.tsx:35-48](src/components/hosted/JoinHostedWorkout.tsx) (`handleJoin`, no displayName); leaderboard fallbacks in [src/components/hosted/HostedWorkoutDashboard.tsx:161-166](src/components/hosted/HostedWorkoutDashboard.tsx). Backend already accepts it: [convex/hostedWorkoutParticipants.ts:12-13](convex/hostedWorkoutParticipants.ts).
  - **Type:** bug / UX
  - **Fix direction:** Collect a display name when a signed-in user joins (prompt/field defaulting to their Clerk name/username), and pass it to `join({ token, displayName })`. Ideally derive a sensible default from the authenticated identity so it's never empty.
  - **Acceptance:** When a signed-in user joins, they appear by a real name (not "Signed-in athlete") in the participants list and on every leaderboard row; two different signed-in athletes are distinguishable.

- [ ] **Duplicate score submissions are allowed (no dedup/upsert)**
  - **What:** The score form can be submitted repeatedly for the same WOD; each submit inserts a new `hostedWorkoutSubmissions` row, producing duplicate leaderboard entries for the same athlete + WOD. Reproduced: two identical "Signed-in athlete / Rx / 5:00" rows on the same WOD. There is no "update my score" concept.
  - **Where:** `/log/$sessionId` → [convex/hostedWorkoutSubmissions.ts:196](convex/hostedWorkoutSubmissions.ts) (`submitForSession`) and [convex/hostedWorkoutSubmissions.ts:225](convex/hostedWorkoutSubmissions.ts) (`submitGuest`), both always `insert`; form has no re-submit guard: [src/components/hosted/HostedScoreForm.tsx:73-86](src/components/hosted/HostedScoreForm.tsx).
  - **Type:** bug
  - **Fix direction:** Enforce one score per athlete (participant, or guest-name) per WOD block: on submit, look up an existing submission (`by_hosted_workout_wod` + participantId/guestName) and patch it instead of inserting, or reject the duplicate. Reflect "update your score" in the form copy.
  - **Acceptance:** Submitting the same WOD twice as the same athlete results in one leaderboard row that reflects the latest score, not two rows.

- [ ] **Each signed-in hosted submission creates a new WOD in the athlete's personal library (and duplicates on re-submit)**
  - **What:** `insertSignedInSubmission` inserts a fresh `wods` doc + `wodResults` doc on every signed-in submission. Reproduced: after two "Fran" submissions, `/wods` "MY WODS" showed two identical user-created "Fran" entries (separate from the benchmark Fran). Re-submitting multiplies them.
  - **Where:** `/log/$sessionId` → [convex/hostedWorkoutSubmissions.ts:152-178](convex/hostedWorkoutSubmissions.ts) (`insertSignedInSubmission` inserts `wods` then `wodResults`).
  - **Type:** bug
  - **Fix direction:** Don't create a brand-new `wods` per submission. Either reuse an existing WOD (link via the block's `wodId` when present, or find-or-create by name/definition), or record the `wodResults` against a shared WOD, and combine with the dedup fix so re-submits update rather than duplicate. Avoid polluting the user's WOD library with one entry per submission.
  - **Acceptance:** Submitting a hosted score does not add duplicate WODs to the athlete's `/wods` library; re-submitting the same WOD does not create additional library entries.
